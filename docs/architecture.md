# Architecture

Chromassist is a Windows-first, local-only desktop application. It uses C#/.NET 10 and WPF because the MVP needs dependable local file access, process execution, Steam-path discovery, and a portable native Windows executable. It does not need a browser runtime or a Rust/TypeScript bridge.

## Boundaries

- `Chromassist.Core`: domain models, TH18 adapter, hashing, image pipeline, THTK process adapter, and thcrap patch writer. It has no WPF dependency.
- `Chromassist.Presentation`: ViewModels, commands, localization, and UI service interfaces. It has no WPF dependency.
- `Chromassist.App`: WPF views and Windows-only implementations for file picking, dialogs, and process launch. Code-behind only wires the ViewModel.
- `Chromassist.Tests`: synthetic fixtures only.

```text
WPF View -> Presentation ViewModel -> Core interfaces -> TH18/THTK/thcrap adapters
```

This boundary allows a future Avalonia, WinUI, or web front end to reuse the Core and Presentation assemblies. The application remains local-first: game files, extracted textures, generated textures, and profile data are never uploaded.

## Data flow

1. Locate `th18.exe` from Steam libraries or a manual file choice.
2. Hash `th18.exe` and `th18.dat`; reject unknown pairs.
3. Inspect the installed thcrap version and Korean patch stack.
4. Run allowlisted `thdat` and `thanm` binaries into a private temporary directory.
5. Mark every extracted texture with a gameplay role. The MVP accepts only `EnemyProjectile`, decodes RGBA8 PNG, transforms only eligible chromatic RGB where alpha is nonzero, and preserves low-chroma neutral pixels.
6. Write an app-owned local repository, `repo.js`, and patch under `thcrap/repos/chromassist/`, then create a new run configuration. Never edit `thpatch-ko.js`.
7. Verify the generated file count, alpha invariants, transparent pixels, neutral pixels, changed opaque-pixel counts, and local `repo.js`, then launch the existing root `thcrap.exe` setup tool only after explicit user action.
8. The user selects the `TH Chromassist (local)` patch in thcrap and lets thcrap generate a persistent shortcut. Normal later launches use that shortcut and do not require Chromassist.

## Optional stage context flow

The user may select one or more PNG, JPEG, or BMP gameplay screenshots in the strength step. The Windows layer decodes a bounded-size copy into memory and passes pixels plus the file name to the presentation layer; it does not retain the source path. The Core compositor places representative enemy-projectile atlas regions over the same background for the original and adjusted views. Background pixels are never transformed, the selected screenshots are never written into the generated patch, and changing the selected background does not change the palette.

This is deliberately a two-level preview: the neutral atlas view exposes the color change itself, while the optional stage view exposes possible foreground/background merging. It is an A/B inspection aid, not a color-vision simulation or proof of benefit.

The shortcut behavior follows the [thcrap file and launcher documentation](https://docs.thpatch.net/en/latest/2_files/): shortcuts generated by `thcrap_configure` invoke `thcrap_loader` with a run configuration and game identifier. The log-based runtime verifier remains a Core diagnostic utility, but the setup flow does not claim runtime texture resolution before the user launches the generated shortcut.

## Fail-closed rules

Unknown game hashes, unknown THTK hashes, incompatible thcrap versions, unexpected texture names or roles, unsupported PNG encodings, or invariant violations stop generation. Original game archives and the existing translation configuration are read-only.
